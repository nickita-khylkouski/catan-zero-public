## 1. Tech Radar

### Python 3.11
- **Category:** Runtime
- **How it's used here:** The package, tests, launchers, search orchestration, data tooling, and trainers are Python. The deployment is pinned to 3.11 because the native wheel is CPython 3.11-specific.
- **What you need to follow:** Use the project virtual environment, never a host's bare `python3`.

### Catanatron + `catanatron_rs`
- **Category:** Simulator
- **How it's used here:** Vendored Catanatron provides the full Python environment; the Rust binding is the production 2-player simulation, native feature, and MCTS-state backend.
- **What you need to follow:** Treat Python/Rust rule and feature parity as a release gate. Do not silently fall back from a requested native path.

### PyTorch
- **Category:** Model and training framework
- **How it's used here:** Entity-token policies, batched leaf evaluation, BC/PPO learning, DDP/FSDP, and BF16 H100 training all use PyTorch.
- **What you need to follow:** Checkpoint metadata controls architecture and hidden-information masking; inference configuration must agree with it.

### NumPy, NPZ, and memmap
- **Category:** Data layer
- **How it's used here:** Self-play decisions are emitted as NPZ shards and converted to streaming memmap corpora for large training runs.
- **What you need to follow:** Preserve schema/provenance fields, group validation by whole game seed, and use fresh output directories.

### `multiprocessing` + EvalServer
- **Category:** Parallel inference
- **How it's used here:** Independent game workers retain CPU parallelism while a per-GPU server batches neural forwards across processes.
- **What you need to follow:** One generator/eval-server belongs to one physical GPU. Threads are not a substitute because Python search/feature work remains GIL-bound.

### pytest + canonical shell gate
- **Category:** Verification
- **How it's used here:** `make test` runs the champion no-op, native feature parity, CLI goldens, and the main suite.
- **What you need to follow:** Performance-only changes still require semantic parity. Search-policy changes additionally require paired H2H gates.

## 2. High Level Design

### System Overview

CatanZero is a research stack for expert-iteration Catan agents. Its mature production track is two-player, no-trade Catan: a Rust simulator supplies states and legal actions, Python Gumbel chance MCTS selects moves, a PyTorch entity-graph policy evaluates leaves, and workers write decision shards. Training turns those shards into a new checkpoint, then paired, seat-swapped statistical gates decide promotion.

The repository also contains a richer 2–4 player Python environment with structured trades, negotiation, replay, and PPO adapters. That surface is closer to the declared `CatanBench-4P-Full-v1` goal, but the current production generator/gate is still the 2-player track.

### Request Flow

```text
Fleet launcher
  → one generator + EvalServer per physical GPU
  → multiprocessing game workers
  → catanatron_rs.Game
  → Python GumbelChanceMCTS
  → native entity/context features
  → batched EntityGraphPolicy forward
  → NPZ decision shards
  → curated memmap corpus
  → train_bc.py via DDP/FSDP
  → paired H2H + SPRT/promotion gate
  → generator/public champion registries
```

## 3. Module Directory

### `/src/catan_zero/search`
- **Owns:** Rust-backed search adapters, Gumbel chance MCTS, neural leaf evaluation, batching, and subtree-reuse support.
- **Entry point:** `src/catan_zero/search/__init__.py`
- **Key files:**
  - `gumbel_chance_mcts.py` — Python tree policy, chance expansion, and sequential halving.
  - `neural_rust_mcts.py` — native feature adapter, cache, batched evaluation, and checkpoint safety gates.
  - `eval_server.py` — cross-process neural forward batching.
- **Connects to:** `rl/entity_token_policy.py`, `rl/gumbel_self_play.py`, and `catanatron_rs`.
- **Contribute here if:** Work changes simulations per second, search semantics, leaf batching, or native integration.

### `/src/catan_zero/rl`
- **Owns:** Environments, self-play game loops, policies, feature schemas, PPO/BC helpers, replay, and league/flywheel state.
- **Entry point:** `src/catan_zero/rl/__init__.py`
- **Key files:**
  - `gumbel_self_play.py` — production Rust-game loop and training-row construction.
  - `entity_token_policy.py` — entity-graph model and checkpoint contract.
  - `multiagent_env.py` — richer Python 2–4 player environment.
- **Connects to:** `search`, `distributed`, vendored Catanatron, and pipeline tools.
- **Contribute here if:** Work changes observations, policies, data targets, training behavior, or full-game simulation.

## 4. Assumption Registry

### [HIGH CONFIDENCE] The production strength ladder is still 2-player/no-trade
- **Evidence:** The canonical generator and promotion tools pass `track=2p_no_trade`; the four-player benchmark runner is not complete.
- **If correct →** Report 2-player and four-player progress separately.
- **If wrong →** Locate a newer external benchmark harness and registry outside this checkout.
- **Verify:** `rg -n "2p_no_trade|NotImplementedError" tools/fleet tools/promotion_gate_runner.py src/catan_zero/benchmark.py`

### [HIGH CONFIDENCE] Hidden-information masking is a checkpoint contract
- **Evidence:** `EntityGraphRustEvaluator` compares requested public observation mode with checkpoint metadata and fails closed.
- **If correct →** Never flip masking only at inference.
- **If wrong →** Audit the checkpoint loader and current training save metadata.
- **Verify:** `src/catan_zero/search/neural_rust_mcts.py` and `src/catan_zero/rl/entity_token_policy.py`

### [HIGH CONFIDENCE] Cross-process evaluation is the supported GPU batching path
- **Evidence:** The generator explicitly rejects the repeatedly slower threaded path and wires EvalServer clients to process workers.
- **If correct →** Scale worker processes and one server per GPU.
- **If wrong →** Re-profile only after Python search/feature work releases the GIL.
- **Verify:** `tools/generate_gumbel_selfplay_data.py` and `src/catan_zero/search/eval_server.py`

### [MEDIUM CONFIDENCE] Eval cache should be disabled for volume self-play
- **Evidence:** Code comments and prior profiles report negligible transpositions but nonzero snapshot hashing/key cost.
- **If correct →** Use `--eval-cache-size 0` for independent self-play generation.
- **If wrong →** A future subtree-reuse/transposition design may create enough hits to justify a bounded cache.
- **Verify:** Add hit/miss telemetry to a representative long run.

### [MEDIUM CONFIDENCE] Independent outcomes currently buy more strength than higher simulation budgets
- **Evidence:** The v9 analysis attributes value overfit to game-label scarcity; rollout-doubling is not yet powered on H100.
- **If correct →** Favor fresh n64 games and targeted restarts after throughput certification.
- **If wrong →** A powered n64-vs-n128 paired gate will show material teacher strength per GPU-hour.
- **Verify:** Run the controlled rollout-doubling experiment in the execution plan.

### [LOW CONFIDENCE] All 24 H100s share the new 8-GPU node topology
- **Evidence:** Only one 8×H100 host was freshly inventoried; older documents describe six 4×H100 boxes.
- **If correct →** Reuse the NUMA/NVSwitch placement plan across three nodes.
- **If wrong →** Keep host-specific worker and NCCL settings.
- **Verify:** Inventory every current host before fleet-wide deployment.

## 5. Before You Start Coding

### Must-Know Patterns
1. **Performance-only versus search-semantic changes** — only the former can use parity alone; the latter need paired strength gates.
2. **Explicit production configuration** — masking, seed ranges, c-scale, temperature, chance handling, and feature path must not rely on parser defaults.
3. **One owner and one canonical path per box** — use alias-based fleet tools, fresh outputs, explicit PIDs, and heartbeat supervision.

### Common Mistakes to Avoid
- ❌ Expose several GPUs to one `--device cuda` generator → ✅ launch one generator/eval-server per physical GPU.
- ❌ Multiply seed claims by worker count → ✅ workers partition `--games`; claim games per GPU only.
- ❌ Treat more correlated rows as more value labels → ✅ measure independent game outcomes and group validation by game.
- ❌ Apply the archived optimization patch bundle wholesale → ✅ land individually tested changes with current-stack benchmarks.
- ❌ Claim the four-player benchmark from two-player results → ✅ keep separate powered evaluation milestones.

## 6. Feature Deep Dive

### Simulation Throughput and Playing Strength

**To contribute here, start with:**
1. `tools/fleet/fleet_launch.sh` — physical GPU/process placement and guarded production configuration.
2. `tools/generate_gumbel_selfplay_data.py` — worker orchestration and manifest contract.
3. `src/catan_zero/rl/gumbel_self_play.py` — game loop and decision rows.
4. `src/catan_zero/search/gumbel_chance_mcts.py` — simulation traversal and search semantics.
5. `src/catan_zero/search/neural_rust_mcts.py` — native features and neural leaf evaluation.

**Before touching this feature:**
- Preserve seed determinism, hidden-information boundaries, forced-action weights, chance spectra, and pre-action feature capture.
- Benchmark complete games/rows/simulations per second, not GPU utilization alone.
- Treat EvalServer batching as numerically equivalent within tolerance, not necessarily bit-identical.
- Use powered paired H2H before promoting D6, subtree reuse, new tree traversal, or changed simulation budgets.

**Likely files you'll modify:**
- `tools/fleet/fleet_launch.sh`
- `src/catan_zero/search/neural_rust_mcts.py`
- `src/catan_zero/search/gumbel_chance_mcts.py`
- `tests/test_fleet_launch_contract.py`
- `tests/test_evaluator_shared_payload.py`
