# CAT-74: 4-Player + Trade Extension Cost Audit

**Status:** Analysis only. No code changed. All claims below are grounded in file:line citations in this worktree (branch `cat-74-4p-audit`) and were spot-verified by direct reading, not grep-inference alone.

**TL;DR:** The codebase is not one monolithic 2p/no-trade system — it is three separate subsystems with wildly different 4p+trade readiness. The production Gumbel-MCTS/Rust-search track (the one actually generating training data today) is genuinely 2p/no-trade at its core search math, but the surrounding featurization and data schema are already 4-player-shaped. A second, currently-unused-for-generation PPO/gym track already has a working n-player negotiation state machine. The single largest blocker is that the production search's value backup is a binary zero-sum negation with no defined meaning for 3+ players, and the production game engine (`catanatron_rs`, an external Rust binding) has never implemented trade as a game action at all — only bank/maritime trade. Recommendation at the end.

---

## 0. Three subsystems, not one (read this first)

Everything below is easier to place once you know which of these three code paths it lives in:

1. **Production Gumbel-MCTS self-play track** — the one generating today's training corpora. Entry point `tools/modal_gumbel_factory.py:197` → `src/catan_zero/rl/gumbel_self_play.py` (`run_worker_games`/`play_one_game`) → `src/catan_zero/search/gumbel_chance_mcts.py` + `src/catan_zero/search/neural_rust_mcts.py` + `src/catan_zero/search/rust_mcts.py`, driven directly by the external `catanatron_rs` Rust binding (`gumbel_self_play.py:665`). `GumbelSelfPlayConfig.track: str = "2p_no_trade"` and `COLORS = ("RED", "BLUE")` are explicit config literals at `gumbel_self_play.py:86,168,170` — a deliberate default, not a hard engine ceiling, but the search math built on top of it (§3) does not currently generalize past 2p regardless of that default.
2. **PPO/gym/negotiation track** — `src/catan_zero/rl/gym_env.py`, `multiagent_env.py`, `negotiation.py`, `self_play.py`. Actively used for BC/PPO baseline training, evaluation ladders (`elo_ladder.py`, `evaluate_openings.py`, `league_orchestrator.py`), and heuristic-bot opponents — **not** the primary data generator. This track is already meaningfully n-player and trade-capable: `ColonistMultiAgentConfig.players: int = 4`, validated against `(2, 3, 4)` (`multiagent_env.py:36,65-66`), and `negotiation.py`'s `ColonistNegotiationState`/`NegotiationOffer`/`TradeSide` (propose/respond/counter/resolve-to-concrete-trade) is fully wired into both `gym_env.py` (lines 250-366) and `multiagent_env.py` (lines 510-604), not dead scaffolding.
3. **A 4-player equivalence/benchmark harness** — `src/catan_zero/engine.py` + `src/catan_zero/adapters/catanatron.py` + `src/catan_zero/rules.py`, used only by `tools/gumbel_search_vs_bot_h2h.py`, `tools/engine_equivalence_sweep.py`, and the `test_environment_contracts`/`test_engine_equivalence`/`test_rules`/`test_schemas` test files. `rules.py:27-28` actually *requires* exactly 4 players (raises otherwise) and `adapters/catanatron.py:464-469` hardcodes all four colors. This is a correctness-checking tool against the legacy Python `catanatron` engine, unrelated to production generation.

The ticket's framing ("the current 2p-no-trade system") really describes subsystem 1 plus the shared featurization layer that subsystem 1 and 2 both draw from. The cost estimate below is scoped to what it takes to bring subsystem 1 (the thing that actually needs to generalize for a public 4p+trade claim) up to par, reusing subsystem 2/3 where possible.

## 1. State representation / 2p-hardcoding in engine and adapters

**Finding: mostly already generic. Cost: SMALL.**

- Robber victim selection — hypothesized as a likely "no-op assumption" in 2p — is **not** hardcoded anywhere. Upstream vendored engine (`vendor/catanatron/catanatron/catanatron/models/actions.py:208-246`, `robber_possibilities`) generates one `MOVE_ROBBER` action per `(tile, enemy_color)` for every color present. Our own action catalog independently rebuilds the same thing generically: `src/catan_zero/rl/action_mask.py:100-102` loops `for victim_color in (None, *player_colors)` over an arbitrary-length `player_colors`. `src/catan_zero/adapters/catanatron.py:311-316,436-441` passes the native `(coordinate, victim)` tuple through unmodified. In 2p games there's naturally at most one non-`None` victim choice — that's a property of the game state, not a code short-circuit.
- `src/catan_zero/engine.py` and `src/catan_zero/schemas.py` use seat integers and tuples of `PublicPlayerState`, generic over player count; no 2p literals found.
- `catanatron_rs`'s Python call surface takes an arbitrary `colors` list at every real call site (`gumbel_self_play.py:665`, `neural_rust_mcts.py:188,267`) via `list(colors)` — only the capability probe (`rust_mcts.py:30`, `catanatron_rs.Game.simple(["RED","BLUE"], seed=0)`) and the `RustMCTSConfig.colors` default (`rust_mcts.py:42`) hardcode 2; both are trivially widened. **Caveat:** the Rust crate's own source is not vendored in this repo (only referenced by path in a comment, `rust_mcts.py:17`, pointing at a local Downloads folder) — whether `catanatron_rs` has an actual N-player-correct implementation underneath (versus just accepting an arbitrary color list but being untested/undefined past 2) was not verifiable from this repo alone. Flagged as `? UNCERTAIN`, not assumed either way.

## 2. Data / shard schema

**Finding: already 4p-shaped where it matters. Cost: SMALL.**

- `src/catan_zero/distributed/schemas.py:38` (`opponent_policy_ids: tuple[str, ...]`) is variadic.
- The actual production shard schema already writes fixed **4-wide** arrays: `gumbel_self_play.py:452-459` builds `final_public_vps`/`final_actual_vps` indexed by the canonical 4-name tuple `PLAYER_NAMES = ("BLUE", "RED", "ORANGE", "WHITE")` (`gumbel_self_play.py:87`), zero-filled for seats not in play. This means 2p-generated shards are already stored in the shape 4p shards would use — no schema migration needed to start writing 3p/4p rows alongside existing 2p ones.

## 3. Featurization / entity-token layout

**Finding: already player-count-generic up to 4. Cost: SMALL.**

- `src/catan_zero/rl/entity_token_features.py:11` defines `PLAYERS = ("BLUE", "RED", "ORANGE", "WHITE")` — a 4-slot canonical tuple, not a self/opponent pair. `_player_tokens` (lines 367-403) builds a fixed `(4, PLAYER_FEATURE_SIZE)` array, looping `for name in PLAYERS` (line 371) and skipping any name absent from the current game (`if idx is None or name not in players: continue`, line 373) — i.e. it's a padded 4-row table that already degrades to 2 populated rows + 2 zero rows for today's 2p games.
- `mask_player_tokens_public` (entity_token_features.py:79-107) masks every non-actor row via a boolean `nonactor` mask over the full `(B, 4, F)` tensor — generic over 1, 2, or 3 opponents, not hardcoded to exactly one.
- The legacy PPO track's own encoder, `graph_history_features.py:169-186` (`_encode_players`), independently caps at 4 the same way (`names = list(players.keys())[:4]`).
- `neural_rust_mcts.py`'s own masking helper, `_mask_players_to_public` (lines 1209-1238), iterates `for color, player in players.items()` generically.

This is the most favorable finding in the audit: the tensor shape work that would normally be the first expensive step of a 4p extension has effectively already been done, likely as an artifact of building against the real 4-color catanatron color space from the start rather than deriving shapes from "2 players."

## 4. Action catalog / trade actions

**Finding: build/road/settlement/city/dev-card/robber/bank-trade only; domestic trade is explicitly and deliberately deferred. Cost: MEDIUM (schema) to LARGE (full combinatorial trade space), with a real warm-start path.**

- `src/catan_zero/rl/action_mask.py`'s `ActionCatalog` (`_action_array`, lines 55-130) enumerates `ROLL`, `DISCARD_RESOURCE` (×5), `BUILD_ROAD`/`BUILD_SETTLEMENT`/`BUILD_CITY`, `BUY_DEVELOPMENT_CARD`, the three dev-card plays, `MOVE_ROBBER` (×tiles × victims), `MARITIME_TRADE` (bank trade only, 60 entries), `END_TURN`. The class docstring (lines 133-140) states outright: *"Domestic trade offers are not represented in this first flat catalog; later trade training should add a separate structured head instead of widening this action dimension with every possible offer tuple."* This is a documented, deliberate scope cut, not an oversight.
- Measured catalog sizes (executed, not estimated): `ActionCatalog(('BLUE','RED')).size == 332`; 3-color `== 351`; 4-color `== 370` (BASE map). Going from 2p to 4p only adds ~38 `MOVE_ROBBER` victim-slot entries (+11%) — everything else in the existing catalog is already player-count invariant.
- Upstream vendored catanatron's native `ActionType` enum (`vendor/catanatron/catanatron/catanatron/models/enums.py:68-106`) includes exactly five domestic-trade types beyond ours: `OFFER_TRADE`, `ACCEPT_TRADE`, `REJECT_TRADE`, `CONFIRM_TRADE`, `CANCEL_TRADE` (lines 99-104), each with documented value encodings (e.g. `OFFER_TRADE` is a 10-resource give/receive freqdeck tuple; `CONFIRM_TRADE` an 11-tuple naming the accepting color). Everything else in the native enum is already present in our catalog.
- **Genuinely good news on warm-start:** `entity_token_features.py:22-41`'s `ACTION_TYPES` (18 entries) and `CATEGORIES` (line 42) already include the lowercase trade-type strings `offer_trade`, `accept_trade`, `reject_trade`, `cancel_trade`, `confirm_trade` and a `"trade"` category — carried over from the richer `ColonistMultiAgentEnv` action vocabulary that track 2's featurizer shares. These are currently dead one-hot columns (production `ActionCatalog` never emits those type strings), but `action_features.py:12-31`'s `_PRIORITY` dict has matching entries too. Both nets that consume this (`entity_token_policy.py`'s `EntityGraphNet`, `xdim_lite_policy.py`'s `XDimLiteNet`/`XDimGraphNet`) compute action logits via a CLIP-style dot product between a state embedding and a *per-action* embedding derived from that action's fixed-width one-hot feature vector — **not** a `Linear(hidden, action_size)` layer or an `nn.Embedding(action_size, ...)` table sized at construction. Verified: `config.action_size` is stored only for shape-validation bookkeeping and is never used to size any weight matrix inside `EntityGraphNet._Module.__init__`. Practical implication: wiring up trade actions that fit inside the *existing* one-hot slots requires no architectural resize — only fine-tuning so the network learns to use columns that have always been zero. A materially larger action space (see next bullet) would still need new feature-schema slots, which *would* require a shape change and full retrain of the action encoder for those slots.
- One inconsistency worth flagging while it's cheap to fix: `graph_history_features.py:22-39`'s own `ACTION_TYPES` list has only `offer_trade`/`accept_trade`/`reject_trade` — missing `cancel_trade`/`confirm_trade` that `entity_token_features.py` has. Minor drift between the two feature-encoding modules, not blocking, but worth reconciling before either is trained on real trade data.
- Branching-factor magnitude: today's flat catalog (332-370 IDs) is the *upper bound over the whole game*, not the legal-action count at a single decision. A `MARITIME_TRADE`-style flattened single-resource-for-single-resource domestic offer catalog would add roughly tens of IDs per potential trade partner (mirroring the existing 60-entry bank-trade block). A faithful multi-resource-bundle propose-trade space (arbitrary give/receive freqdecks × counterparty × accept/reject/counter) is combinatorially larger — orders of magnitude — which is exactly why the catalog docstring already recommends a separate structured head rather than widening this flat dimension. This tracks with how Cuayáhuitl et al.'s negotiation module and Keizer et al. (EACL 2017) approached the trade subtask: both used a dedicated negotiation-specific module/policy rather than folding trade into a flat move-selection action space (see §7).

## 5. Value / self-play semantics (the real structural blocker)

**Finding: the production search's completed-Q backup is an unguarded binary zero-sum negation; the evaluator's mirror of it is guarded to silently no-op for 3+ players. Cost: LARGE.**

- `src/catan_zero/search/gumbel_chance_mcts.py:714-715` (`_completed_q`): `root_to_act = str(node.game.current_color()) == str(node.root_color); sign = 1.0 if root_to_act else -1.0`. Every non-root color collapses into one undifferentiated "opponent" role, and the assumption `value(any-other-seat) == -value(root)` is applied unconditionally — this is the completed-Q/backup math for the entire Gumbel search, and it is mathematically well-defined only under 2-player zero-sum symmetry. There is no player-count guard here at all, so it would silently run (and silently produce meaningless numbers) if pointed at a 3-4p game today.
- `src/catan_zero/search/neural_rust_mcts.py` mirrors this at the evaluator level, but *does* guard it: `if acting_color != str(root_color) and len(tuple(colors)) == 2: value = -value`, repeated identically at lines 249, 340, 467, and 714. For `len(colors) != 2` this condition is simply `False` — the raw, unflipped, acting-seat-perspective value is returned with no correction and no error. Combined with (1), a 3-4p run today would neither crash nor produce a valid multiplayer value — it would silently produce garbage that looks like a number.
- `src/catan_zero/search/rust_mcts.py:315-326` has the identical binary sign flip in the legacy PUCT search, with its own comment acknowledging the limitation: "Zero-sum/paranoid backup for 2p and early multiplayer work" (line 323) — i.e. even where the authors anticipated multiplayer, they explicitly scoped it as a paranoid (worst-case opponent) approximation, not a "true" N-player value.
- The one piece that *is* already N-player-safe on its own: terminal backup is a plain binary win/loss (`gumbel_chance_mcts.py:856-858`, `rust_mcts.py:216`, `neural_rust_mcts.py:1369-1373`, `1.0 if winner == root_color else -1.0`). But it feeds into the same binary-negation completed-Q logic at every interior node, so the overall backup chain is not N-player-correct end to end.
- **What fixing this actually requires:** a redefined interior-node value convention for N>2 — options include (a) a paranoid/pessimistic reduction (worst-case over opponents, extending the existing `rust_mcts.py:323` comment's own framing, likely the cheapest to implement since the sign-flip machinery already has a "paranoid" precedent to extend rather than invent), (b) a per-seat value vector with a defined reduction rule for backup, or (c) a scalar "P(root wins)" trained target that sidesteps zero-sum symmetry entirely. Any of these requires rewriting `_completed_q`, `_select_action`, and all four `neural_rust_mcts.py` sign-flip sites consistently, plus retraining the value head against whatever new target semantics are chosen — this is the single largest line item in the whole audit, both in code-reasoning risk (subtle backup bugs are exactly the class of bug this project has repeatedly found expensive post-hoc — see the lazy-chance depth>0 ROLL backup bug and the leak-fix precedent) and in required retraining.

## 6. Trade as an intra-turn sub-decision, and search-tree structure

**Finding: the search tree's node/action model is not the blocker for representing trade as a sequence of ordinary decision nodes — DISCARD is already proof of that pattern. The actual blocker is that the production Rust engine has never implemented trade as a game action. Cost: LARGE, and it's an engine (Rust) build, not a Python search restructure.**

- Turn identity in the search is never hardcoded as fixed alternation — every node re-derives whose turn it is via `game.current_color()` (`gumbel_chance_mcts.py:427,714`; `rust_mcts.py:191,315`; `gumbel_self_play.py:483,695`), with an explicit comment (`gumbel_self_play.py:636-638`) noting the acting evaluator is reassigned every decision, not fixed at game start. Robber-victim chance weighting (`gumbel_chance_mcts.py:1350-1408`, `move_robber_victim_outcome_weights`) indexes by `colors.index(str(victim_name))` over an arbitrary-length `colors` tuple. Turn rotation and chance-node structure already generalize to N-ply rotation; they ride on top of the (broken, per §5) value semantics, but the *tree shape* itself is not the problem.
- The legacy Python `catanatron` adapter's own phase state machine (`src/catan_zero/adapters/catanatron.py:135-160`, `_phase_for_state`) already models `ActionPrompt.DECIDE_TRADE` and `ActionPrompt.DECIDE_ACCEPTEES` as ordinary forced single-actor decision nodes, mapped to `Phase.TRADE_RESPONSE` (lines 156-157) — structurally identical to how it already handles `DISCARD` (multiple players get a forced decision node, one at a time, before the roller's main phase resumes). This confirms that "propose trade → wait for named respondents → resolve" fits the existing node model (one atomic legal action per node, next actor re-derived from state) without a fundamental restructure of the search's Python side.
- The actual gap: `catanatron_rs` — the only engine the production Gumbel search is wired to — has no trade-negotiation prompt at all; grepping for `DECIDE_TRADE`/`OFFER_TRADE`/any trade-prompt handling near `rust_mcts.py`/`gumbel_chance_mcts.py`/`neural_rust_mcts.py` found only `MARITIME_TRADE` (bank trade) references (`neural_rust_mcts.py:1260,1394`). The Rust crate's own source was not available in this repo to confirm definitively whether it has any unexposed trade scaffolding (flagged `? UNCERTAIN`, not assumed), but nothing in the Python call surface indicates one exists. Building this is therefore primarily a Rust game-engine feature (implement the domestic-trade phase and prompts, mirroring what the vendored legacy Python engine already has working), not a Python MCTS redesign — likely the single largest and least Python-reasoning-verifiable item in the whole audit, since it requires work in a codebase not present here.

## 7. Literature sanity check (already surveyed in the master plan, cited here for this audit's context)

Per `docs/plans/CATAN_ZERO_MASTER_PLAN.md:118` and `docs/reviews/review-7-full.md:62`, the existing 4p+negotiation literature gives an external calibration point for how much dedicated machinery the trade subtask specifically required elsewhere:

- **Cuayáhuitl et al.** (2015/2017, arXiv:1511.08099): a dedicated DQN negotiation module reached ~82% win rate vs humans on the negotiation subtask — negotiation was built and evaluated as its own module, not folded into the main move-selection policy.
- **Keizer et al.** (EACL 2017): a negotiation-only DRL system beat humans 81.8% at the trade subtask alone.

Both precedents support the finding in §4/§6 above: prior work treated trade as a separately-scoped module/policy rather than a naive expansion of a flat move-action space, which matches this codebase's own `ActionCatalog` docstring recommendation ("a separate structured head instead of widening this action dimension with every possible offer tuple," `action_mask.py:138-140`). This is a point in favor of building any future trade support as its own component rather than trying to retrofit it into the existing flat catalog end to end.

## 8. Reward / gate / league N-player assumptions

**Finding: mixed. Cost: SMALL-MEDIUM.**

- `src/catan_zero/rl/league.py:163-178` (`record_match`) is hardcoded strictly pairwise: `record_match(agent_a_id, agent_b_id, a_score)` with `a_score` documented as "[0,1], 1 = a won" (line 164). `payoff_matrix()` (lines 187-207) builds an N×N matrix, but every cell comes from single-pair game records — it is a matrix *of* pairwise results, not a genuine N-player free-for-all rating model. `sample_opponent`'s PFSP weighting (lines 232-248) likewise uses a single pairwise win rate. This would need either a real multiplayer rating system (placement-based Elo/WHR generalization) or a redefinition of `a_score` as normalized placement to keep reusing the existing pairwise math.
- `src/catan_zero/rl/promotion_gate.py:14-32` (`decide_promotion`) is itself agnostic — it compares two floats named `win_rate` and has no player-count logic baked in. It would tolerate an N-player-derived win rate with zero changes; the pairwise-ness lives entirely upstream in whatever computes that `win_rate` (the H2H gate tooling, noted elsewhere in project history as untracked in git on the training host, not present in this repo to audit directly).
- `src/catan_zero/rl/reanalysis.py:22-27` (`flatten_episode_for_reanalysis`) and `src/catan_zero/rl/ppo_reward_shaping.py:123-174,251-300` (`catan_potential`, `extract_public_state`) are already N-player-generic — both iterate per-seat/per-player-name with no opponent-count assumption. `policy_pool.py` and `hybrid_policy.py` are seat-count-agnostic dispatch utilities.

## What (if anything) to change now, cheaply, while it's still Phase A

Two items in this audit are cheap enough, and calcify enough of the rest, that they're worth doing now rather than deferring, independent of when (or whether) 4p+trade is actually built:

1. **Reconcile the `ACTION_TYPES` drift between `entity_token_features.py` and `graph_history_features.py`** (§4) — a few missing trade-type strings in one of the two files. Trivial now; a real correctness bug waiting to happen the moment either featurizer is trained on real trade-labeled data, and exactly the kind of silent, hard-to-attribute drift this project has paid for before (see the CLI-default-override and doc-vs-code drift precedents from earlier audits).
2. **Turn the silent no-op in `neural_rust_mcts.py`'s `len(tuple(colors)) == 2` guard (§5) into an explicit assertion or NotImplementedError for `len(colors) != 2`.** Right now, pointing any of this code at a 3-4p game produces no crash and no warning — just quietly wrong values. This is a one-line, zero-risk change (it only changes behavior for a code path nothing currently exercises) that converts a future silent-corruption failure mode into a loud one, which is exactly the kind of guard this codebase's own hygiene audits have flagged as missing elsewhere.

Everything else — the value-backup redefinition (§5), the Rust engine trade-prompt implementation (§6), the league/PFSP generalization (§8), and the structured trade action head (§4) — is genuine, non-trivial engineering (LARGE items are measured in weeks, not days, and one of them, the Rust engine work, requires access to a codebase not present in this repo) and should stay a **deliberate Stage 3 scope decision**, not something to build opportunistically now. The infrastructure that *would* calcify around a 2p-only assumption (token layout, data schema, victim selection, turn rotation, reward shaping) turns out to already be player-count generic — the parts that are genuinely 2p-locked (search backup math, the trade action space, the missing engine feature) are exactly the parts too large to build "cheaply" as a side effect of current work anyway. Recommendation: **defer the LARGE items to Stage 3 as planned, but land the two small items above now** so the eventual Stage 3 decision isn't also debugging silent corruption that was cheap to prevent today.

**Re-audit trigger list for whenever Stage 3 is revisited:** re-check whether `catanatron_rs` has gained trade-prompt support upstream in the interim; re-check the `ACTION_TYPES` catalogs haven't drifted further; re-check whether the "Deduction tracker" ticket's exact-deduction argument (2p-specific, "no third parties") has been implemented in a way that assumes 2p, since that would become a second LARGE item to unwind alongside the ones here.
